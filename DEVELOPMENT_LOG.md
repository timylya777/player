Этот файл служит долгосрочной памятью для будущих сессий с AI-помощниками (Antigravity/Claude/Gemini). Он содержит общую информацию о проекте, историю обращений пользователя, выполненные действия и текущую архитектуру, чтобы любой новый агент мог мгновенно войти в контекст.

## 🌌 Общая информация о проекте

**Nebula Player** (Nebula Music Player & Audio Visualizer) — это мультиплатформенный музыкальный плеер с неоновой эстетикой (Synthwave/Cyberpunk), визуализацией звуковых частот, встроенными плейлистами, системой аккаунтов и караоке-текстами (LRC).

### Технологический стек:
* **Фронтенд**: Ванильный HTML5, CSS3, JavaScript (ES6+).
* **Среда выполнения (Десктоп)**: Electron (для Windows). Сборка портативной версии через `electron-builder` в папку `dist-win`.
* **Среда выполнения (Мобильные)**: Capacitor (поддержка Android/iOS).
* **Бэкенд-прокси**: Python HTTP-сервер на порту `9001` (запуск `python yt_proxy.py`), обеспечивающий:
  * Поиск по YouTube с помощью утилиты `yt-dlp`.
  * Стриминг аудио через проксируемый поток `/api/play` для обхода ограничений CORS.
  * Автоматический парсинг субтитров песен на русском, английском и украинском языках (преобразует YouTube-субтитры в формат LRC/таймингов для караоке) через `/api/lyrics`.

#### 2. Исправление бага с заглушкой вместо караоке-текста
* **Запрос пользователя**: *«нужно исправить баг с тем-что при загрузке трека в вкладке текста отображается заглушка»*
* **Диагностика бага**:
  * При создании YouTube-треков в `www/app.js` им присваивалось свойство `track.youtubeId`. Однако функция `loadLyricsForTrack(track)` проверяла `track.videoId`. Из-за этого несовпадения плеер пропускал этап скачивания реального текста с прокси и сразу переключался на генерацию космической заглушки («Настройка телеметрической связи...»).
  * При загрузке трека сначала мгновенно выводилась заглушка, и только потом шёл асинхронный запрос к серверу, что вызывало визуальное мерцание.
  * При буферизации аудио событие `onloadedmetadata` повторно вызывало загрузку текстов, создавая дублирующие запросы (race condition).
* **Выполненные изменения**:
  * **[`www/app.js`](file:///D:/.code/music-player/www/app.js#L1096-L1141)**: Переписана функция `loadLyricsForTrack`. Теперь она безопасно ищет ID (`track.videoId || track.youtubeId`), мгновенно выводит красивое пульсирующее сообщение `"🌌 Загрузка текста песни..."` во время сетевого запроса, кэширует успешно скачанный текст в `track.lyrics` и блокирует повторные запросы во время активного скачивания через `track.isFetchingLyrics`.
  * **[`www/styles.css`](file:///D:/.code/music-player/www/styles.css#L1717-L1725)**: Добавлено описание анимации `@keyframes pulse` для плавного изменения прозрачности при загрузке.

  * Прямой запуск команд в консоли Windows пользователем полностью свободен от ограничений песочницы и работает молниеносно.
* **Действие**:
  * Агент корректно и безопасно завершил все фоновые процессы в песочнице, чтобы освободить порты `9000` и `9001`.

* **Запрос пользователя**: *«Uncaught TypeError: Cannot read properties of null (reading 'addEventListener') at app.js:2367:14»* и *«а ещё не работает текст песни там всё равно заглушка»*

#### 5. Обход блокировки HTTP 429 (Too Many Requests) со стороны YouTube при загрузке субтитров
* **Запрос пользователя**: *«на сайте поиск отавлился а текст несмотря на наличие субтитров у песни не отображается...»*
* **Диагностика бага**:
  * Прямой запрос к URL субтитров (timedtext) из Python через `urllib.request` блокировался YouTube с ошибкой **`HTTP Error 429: Too Many Requests`**, так как Google видит в этом бота.
  * Из-за этого `/api/lyrics` возвращал `500 Internal Server Error`, и тексты песен с YouTube никогда не подгружались.
* **Выполненные изменения**:
  * **[`yt_proxy.py`](file:///D:/.code/music-player/yt_proxy.py#L185-L248)**: Переписана функция `_handle_lyrics`. Теперь она использует саму утилиту `yt-dlp` в режиме `--skip-download --write-subs --write-auto-subs` для скачивания субтитров во временную папку операционной системы (`tempfile.gettempdir()`) в формате `json3`. Это на 100% обходит блокировку 429 (так как `yt-dlp` умеет имитировать клиентов и обходить защиту Google). Скачанный файл парсится и сразу же корректно удаляется из ОС.
  * Добавлена гибкая фильтрация языков (приоритет: `ru` -> `en` -> `uk`), поддерживающая как оригинальные, так и автопереведенные региональные субтитры.

#### 15. Локальное кэширование и защита избранных треков от авто-удаления

#### 19. Исправление пустого зазора (полоски) сверху и окончательное исправление названий кастомных плейлистов в сайдбаре
* **Запрос пользователя / Проблема**: *«сверху в интерфейсе появилась пустая полоска, изначальная ошибка не была исправлена»*
* **Диагностика и исправления**:
  * **Пустой зазор сверху**: В Electron-окружении блок `.custom-titlebar` имел свойство `position: relative`, занимая 34px в обычном потоке, в то время как контейнер `.app-layout` имел `margin-top: 34px`. Из-за этого `.app-layout` сдвигался еще на 34px вниз, оставляя пустой 34px-зазор (полоску) между заголовком и основным интерфейсом. Свойство `.custom-titlebar` переведено в `position: fixed; top: 0; left: 0; right: 0;`, что прижало заголовок к верхнему краю экрана и восстановило бесшовную стыковку с `.app-layout`.
  * **Иконки в плейлистах**: Зафиксировано и закоммичено изменение в `renderCustomPlaylistsNav()`, меняющее `.innerText` на `.innerHTML` для элементов ссылок плейлистов. Это окончательно решило проблему с отображением сырых тегов `<i class="...">` вместо красивых Phosphor-иконок нот в боковой панели.
